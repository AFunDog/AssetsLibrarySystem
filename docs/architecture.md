# 素材管理系统方案

## 1. 项目目标

本项目用于搭建一个具备扩展能力的素材管理系统，覆盖以下对象：

- 文本素材
- 图片素材
- 视频素材
- 音乐素材

当前阶段以基础架构为主，但边界已经调整为：

- 素材管理主流程留在 `Avalonia/.NET`
- `Python FastAPI` 仅作为大模型 HTTP 服务

RAG、召回和自然语言搜索仍以架构预留为主，重点是先把桌面端与模型网关的职责划清楚。

## 2. 设计原则

- 桌面端和模型网关边界明确
- 分层明确，避免入口文件膨胀
- 先抽象能力边界，再逐步填充实现
- 检索链路与 RAG 设计参考 `D:\GitRepository\RenderTest\test2.py`

## 3. 参考能力拆解

`test2.py` 已经体现出后续搜索能力的核心链路：

1. 为素材描述生成向量
2. 使用向量索引做粗召回
3. 使用 reranker 做精排
4. 持久化索引与元数据
5. 面向自然语言查询返回排序结果

因此，本项目后续需要至少具备以下模块：

- `tagging`：素材自动打标与摘要生成
- `indexing`：向量生成、索引构建、索引更新
- `search`：关键词检索、向量召回、重排序
- `rag`：检索上下文拼装与答案生成

当前阶段只保留这些模块的代码位置，不写真实实现。

## 4. 总体架构

### 4.1 Avalonia 桌面端

桌面端承担素材管理主流程，包括：

- 素材库目录登记
- 文件系统扫描与本地仓储
- 素材详情、状态、人工校对
- 检索任务、打标任务、索引任务的编排入口
- 调用 Python 模型网关的桌面工作流

当前主界面以工作台形式展示：

- 左侧：素材库与本地工作流
- 中间：素材队列与当前目录上下文
- 右侧：选中素材详情与模型网关能力面

### 4.2 Python 模型网关

Python 后端建议继续使用 `FastAPI`，但只保留模型服务相关分层：

- `api`
  - 暴露模型 HTTP 接口
  - 处理健康检查、能力清单、生成请求
- `application`
  - 编排 provider 调用
  - 处理 prompt 与 provider 配置
- `schemas`
  - 模型服务请求/响应结构
- `core`
  - 配置、prompt、provider 解析

当前不再在 Python 端实现：

- 素材库管理
- 目录浏览
- 素材元数据维护
- 本地文件扫描

### 4.3 Web 前端骨架

`src/frontend` 仍可保留为后续 Web 方案预留，但当前不是主入口，也不承载素材管理现状。

## 6. 核心领域对象

后续建议围绕以下对象扩展：

### 6.1 Asset

统一素材实体，至少包含：

- `id`
- `name`
- `type`
- `path`
- `size`
- `mime_type`
- `tags`
- `description`
- `status`
- `created_at`
- `updated_at`

### 6.2 Search Request

用于承接自然语言检索与过滤条件：

- 原始查询文本
- 素材类型过滤
- 标签过滤
- 排序方式
- 分页参数

### 6.3 Knowledge Fragment

用于 RAG 阶段组织可引用上下文：

- 素材描述
- 标签集合
- 解析出的结构化摘要
- 外部补充知识

## 7. 后续功能路线

### 阶段一：桌面端基础素材管理

- 扫描素材目录
- 建立本地素材元数据
- 提供桌面端素材列表与详情视图

### 阶段二：模型调用与索引编排

- 桌面端通过 Python 网关抽取文本描述
- 生成标签建议
- 建立向量索引任务入口

### 阶段三：搜索与 RAG

- 关键词检索
- 向量召回
- rerank 精排
- 检索增强回答
