# 素材管理系统方案

## 1. 项目目标

本项目用于搭建一个具备扩展能力的素材管理系统，覆盖以下对象：

- 文本素材
- 图片素材
- 视频素材
- 音乐素材

当前阶段只建设基础架构，不实现真实业务功能。重点是把后续要落地的模块边界提前划清楚。

## 2. 设计原则

- 前后端分离，便于并行开发
- 分层明确，避免入口文件膨胀
- 先抽象能力边界，再逐步填充实现
- 检索链路与 RAG 设计参考 `D:\GitRepository\RenderTest\test2.py`

## 3. 参考能力拆解

`test2.py` 已经体现出后续搜索能力的核心链路：

1. 为素材描述生成向量
2. 使用向量索引做粗召回
3. 使用 reranker 做精排
4. 持久化索引与元数据
5. 面向自然语言查询返回排序结果

因此，本项目后续需要至少具备以下模块：

- `tagging`：素材自动打标与摘要生成
- `indexing`：向量生成、索引构建、索引更新
- `search`：关键词检索、向量召回、重排序
- `rag`：检索上下文拼装与答案生成

当前阶段只保留这些模块的代码位置，不写真实实现。

## 4. 总体架构

### 4.1 后端

后端建议使用 `FastAPI`，采用以下分层：

- `api`
  - 暴露 HTTP 接口
  - 处理请求参数、响应结构
- `application`
  - 编排业务用例
  - 调用仓储、搜索服务、打标服务
- `domain`
  - 定义素材实体、标签、搜索请求等核心模型
- `infrastructure`
  - 对接文件系统、数据库、向量索引、模型推理组件
- `schemas`
  - API 传输模型
- `core`
  - 配置、常量、基础依赖

### 4.2 前端

前端建议使用 `Vue 3 + TypeScript + Vite`，承担以下职责：

- 素材列表浏览
- 素材详情查看
- 标签与描述展示
- 搜索输入与结果呈现
- RAG 问答入口
- 后台任务状态展示

当前阶段只放一个仪表盘式首页，展示系统模块与后续能力位置。

## 6. 核心领域对象

后续建议围绕以下对象扩展：

### 6.1 Asset

统一素材实体，至少包含：

- `id`
- `name`
- `type`
- `path`
- `size`
- `mime_type`
- `tags`
- `description`
- `status`
- `created_at`
- `updated_at`

### 6.2 Search Request

用于承接自然语言检索与过滤条件：

- 原始查询文本
- 素材类型过滤
- 标签过滤
- 排序方式
- 分页参数

### 6.3 Knowledge Fragment

用于 RAG 阶段组织可引用上下文：

- 素材描述
- 标签集合
- 解析出的结构化摘要
- 外部补充知识

## 7. 后续功能路线

### 阶段一：基础素材管理

- 扫描素材目录
- 建立素材元数据
- 提供素材列表与详情接口

### 阶段二：素材打标与索引

- 抽取文本描述
- 自动生成标签
- 建立向量索引

### 阶段三：搜索与 RAG

- 关键词检索
- 向量召回
- rerank 精排
- 检索增强回答
